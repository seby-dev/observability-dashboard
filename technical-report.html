01

Executive Summary

What this system does, why it was built, and what was achieved.

The Observability Dashboard is a full-stack internal tool built to provide real-time visibility into the operational health of organist_bot — an automated gig-finding system that scrapes music job listings, filters them by configurable criteria, and notifies the user via Telegram.

Because organist_bot runs autonomously on a schedule, understanding what happened during each run — how many gigs were found, how fast it ran, whether any errors occurred, and which filters rejected the most opportunities — was previously only possible by manually reading raw Google Sheets log rows. This dashboard transforms that raw data into actionable metrics, interactive charts, and alert-ready health indicators.

Problem Statement

Raw log data stored in Google Sheets is verbose and unstructured. Without aggregation, answering even simple operational questions requires significant manual effort:

  • Has the bot been running faster or slower recently?
  • What percentage of runs encountered errors this week?
  • Which filter configuration is rejecting the most gigs on average?
  • How does today's listing volume compare to the past month?

Solution

A FastAPI backend syncs log data incrementally from Google Sheets into a local SQLite database. Purpose-built SQL aggregate queries — including window functions for cumulative metrics — power a set of REST API endpoints. A React/TypeScript frontend renders these metrics as interactive area charts, KPI cards, filterable run tables, and drill-down detail pages, all auto-refreshing every 60 seconds.

Key Capabilities Delivered

  • Incremental sync — Only new Google Sheets rows are fetched on each sync cycle, keeping the API fast and quota-efficient.
  • Multi-sheet tab sync — Automatically discovers and syncs all rotated log tabs (Logs, Logs 2, Logs 3, …) created when organist_bot hits the 900k-cell threshold. Each tab is tracked independently in sync_state.
  • Content-based deduplication — Rows are deduplicated by (project_id, run_id, timestamp, module, function, line) rather than sheet row number, so clearing and re-using a sheet never causes silent data loss.
  • Cumulative health metrics — Warning and error rates are computed as running percentages over all historical runs, matching the formula used in summary KPI cards.
  • Time-series charts — Run speed, health rate, and listing volume all support flexible date ranges (3H to 3M) plus a custom date-picker.
  • Filter rejection breakdown — Aggregated average rejections per filter per run, visualised as a time-series line chart. AvailabilityFilter block and only modes appear as separate lines; legacy BookedDateFilter data is remapped automatically.
  • Run detail drill-down — Per-run phase timing, funnel counts, JSON log viewer, and per-filter gig rejection tables sorted by fee.
  • Dark / light theme toggle — Persisted to localStorage, applied via CSS custom properties.
  • Telegram alerts — Warning/Error/Critical logs trigger real-time alerts after each sync.
  • Auto-scheduled sync — APScheduler background job runs every N minutes (configurable via .env).

Data Flow

The system operates on a pull-sync-serve loop:

  1. organist_bot appends structured log rows to Google Sheets during each run (scrape → filter → notify phases).
  2. The scheduler fires every N minutes and calls sync_project(), which first calls _discover_log_sheets() to find all rotated log tabs (Logs, Logs 2, Logs 3, …), then reads only new rows from each tab (row number tracked per sheet in sync_state).
  3. New rows are parsed and bulk-inserted into SQLite via INSERT OR IGNORE (idempotent by content key (project_id, run_id, timestamp, module, function, line) — resilient to sheet resets).
  4. After each sync, process_alerts() queries for any unsent WARNING/ERROR/CRITICAL logs and dispatches Telegram messages.
  5. The React frontend polls each metric endpoint every 60 seconds via TanStack Query. Charts respond to date-range controls through URL query params.

Design Decisions & Rationale

Why SQLite instead of Postgres?

The workload is read-heavy, single-machine, and low-concurrency (one writer: the sync job; many readers: dashboard queries). SQLite eliminates operational overhead — no server process, no connection pooling, no credentials. aiosqlite wraps it in an async interface compatible with FastAPI's event loop, giving the ergonomics of a "real" async database without the cost.

Why Google Sheets as the source of truth?

organist_bot already wrote structured logs to Sheets. Rather than requiring the bot to be re-architected to push to a database directly, the dashboard treats Sheets as an append-only log store and syncs incrementally. This keeps the bot simple and the dashboard independently deployable.

Why React Query (TanStack Query)?

Raw fetch calls in components create duplication and race conditions. TanStack Query provides automatic caching, deduplication, background refresh, and loading/error states out of the box — turning what would be 40 lines of state management into a single useQuery call.

Why CSS custom properties for theming instead of a CSS-in-JS library?

CSS variables (--bg-card, --indigo, etc.) toggled by a data-theme attribute on <html> give zero-JS theming with smooth CSS transitions — no runtime overhead, no hydration issues, and full browser-native performance.

Complete Module & File Breakdown

What every file does, the key logic inside it, and why it was written that way.

Backend — backend/config.py

Loads configuration from two sources: projects.yaml (project definitions) and .env (secrets). Uses Python @dataclass for structured, type-annotated config objects instead of plain dicts.

  • ProjectConfig — holds id, name, sheets_id, sheet_name, credentials_file for one monitored project.
  • Settings — top-level config: list of projects, Telegram credentials, sync interval, DB path.
  • PROJECT_MAP — a module-level dict[str, ProjectConfig] for O(1) lookup by project ID across all request handlers.
💡
Design choice: module-level singleton

Config is loaded once at import time via settings = load_settings(). FastAPI imports happen once per process, so this is safe and avoids repeated YAML parsing on every request.

Backend — backend/db.py

Owns the SQLite schema, connection management, and all data-access queries. It is the only module that ever calls aiosqlite.connect().

Schema

TablePurposeKey columns
logsEvery log line from every runproject_id, run_id, timestamp, level, message, details (JSON), alert_sent
sync_stateTracks last synced row per project per sheet tabproject_id + sheet_name (composite PK), last_row, last_synced_at

Key Patterns

  • Context manager connection — @asynccontextmanager async def get_db() wraps every connection in a try/finally-equivalent, ensuring the connection is always closed even if a query raises.
  • row_factory = aiosqlite.Row — enables row["column_name"] access instead of positional indexing, which makes query results self-documenting.
  • INSERT OR IGNORE — the unique dedup index on (project_id, run_id, timestamp, module, function, line) makes every insert idempotent by log content rather than sheet position. Re-syncing the same rows, or syncing from a reset sheet, is safe — existing rows are silently skipped.
  • _RUNS_CTE — a shared SQL Common Table Expression that pivots per-log-line rows into per-run summary rows using conditional aggregation (MAX(CASE WHEN ...)). Reused by both list_runs() and get_run() to keep the SQL in one place.
  • Indices — five indices cover the access patterns: (project_id, run_id), (project_id, timestamp), (project_id, level), (project_id, alert_sent, level), and the content-based dedup index above. Almost every query is an index scan, not a full table scan.

Backend — backend/sheets.py

Handles all communication with the Google Sheets API, transforming raw row arrays into structured dicts ready for SQLite insertion.

  • _build_service() — creates a Sheets API service client using a service account JSON key file. cache_discovery=False prevents writing discovery docs to disk in serverless-style environments.
  • _discover_log_sheets() — calls the Sheets API's spreadsheets.get method to list all sheet tabs, then returns those whose titles match Logs, Logs 2, Logs 3, etc. organist_bot rotates to a new tab when a sheet approaches the 900k-cell limit; discovery ensures the dashboard automatically picks up new tabs without config changes.
  • _row_to_dict() — maps a raw list of cell values to a named dict. Uses a defensive inner function _get(index) that returns "" for any missing column, making the parser resilient to short rows (common when optional fields are blank).
  • Incremental fetch — last_row is read from sync_state (keyed by (project_id, sheet_name)) before every sync. The API range is A{last_row+1}:I, so only genuinely new rows are transferred. Each tab is tracked independently, so a new tab starts from row 1 without affecting the cursor of older tabs.

Backend — backend/metrics.py

The analytical core of the backend. Contains five async functions, each returning structured data for a specific chart or KPI section. No HTTP logic lives here — it is purely a data-transformation layer.

get_overview()

Executes eight separate SQL queries to populate the two KPI rows on the dashboard. Uses Python's statistics module for in-process mean and median calculations on speed data (Python is fast enough for lists of a few hundred numbers; no need for SQL here).

The most subtle part: runs today requires the caller to pass a timezone string (tz), which is used with Python's ZoneInfo to compute the correct UTC midnight for "today" in the user's local timezone. The frontend sends Intl.DateTimeFormat().resolvedOptions().timeZone as the tz query parameter so the backend always works in the user's local day boundary.

get_speed_series()

Returns per-run timing breakdowns. Uses a template SQL string (SELECT_COLS) with f-string conditional clauses to handle two modes: time-windowed (when since is provided) and limit-based (when no date range is selected). The results are reversed after fetching so the chart always receives chronological order.

get_health_series() — The Most Important Function

This function produces the cumulative warning/error rate series. It is the most technically sophisticated piece of code in the project, and its design was the result of a deliberate debugging session to fix a metric mismatch. 

-- Two-stage CTE with SQL window functions
WITH per_run AS (
    SELECT
        run_id,
        MIN(timestamp) AS started_at,
        MAX(CASE WHEN level = 'WARNING' THEN 1 ELSE 0 END) AS has_warning,
        MAX(CASE WHEN level IN ('ERROR', 'CRITICAL') THEN 1 ELSE 0 END) AS has_error
    FROM logs
    WHERE project_id = ?       -- ← NO time filter here — must see ALL runs
    GROUP BY run_id
),
with_rates AS (
    SELECT
        started_at,
        ROUND(
            SUM(has_warning) OVER (ORDER BY started_at
                ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
            ) * 100.0 / ROW_NUMBER() OVER (ORDER BY started_at),
        1) AS warn_rate_pct,
        ...
    FROM per_run
)
SELECT started_at, warn_rate_pct, error_rate_pct
FROM with_rates
WHERE started_at >= ?   -- ← time filter applied AFTER window computation
ORDER BY started_at ASC
🔑
Why the time filter must be in the outer SELECT, not the CTE

If WHERE started_at >= ? were placed inside per_run, the window functions would only see runs from that date onwards. The cumulative sum would restart at zero, making the last chart point disagree with the overview summary. By filtering after the window computation, we show a sub-range of the global trajectory — so the final data point always equals the all-time rate shown in the summary card.

get_listings_windows()

Buckets runs into N-hour windows using integer epoch arithmetic in SQLite: (CAST(strftime('%s', ts) AS INTEGER) / window_secs) * window_secs. This computes the start of each window without needing any date functions on the Python side.

get_filter_breakdown()

Reads raw JSON strings from the details column, parses them in Python, and accumulates rejection counts per filter name. _shorten_filter_name() normalises raw filter __repr__ strings for display: FeeFilter(min_fee=100) becomes FeeFilter; AvailabilityFilter(mode='block', periods=2) becomes AvailabilityFilter (block); AvailabilityFilter(mode='only', …) becomes AvailabilityFilter (only) — so the two modes appear as separate chart lines. Legacy BookedDateFilter entries (from older run logs) are remapped to AvailabilityFilter (block) for visual consistency.

Backend — backend/alerts.py

Sends Telegram messages for any WARNING/ERROR/CRITICAL logs that have not yet been delivered (alert_sent = 0). Key design decisions:

  • Stop-on-failure — if a Telegram send fails (rate limit, network error), the loop breaks immediately. This prevents flooding the retry queue with messages that will all fail for the same reason.
  • Mark-after-send — alert_sent is only set to 1 after the HTTP call succeeds. This ensures at-least-once delivery: if the process crashes mid-loop, the unsent alerts will be re-attempted on the next sync.
  • Graceful no-op — if TELEGRAM_TOKEN is not configured, the function returns False immediately without raising. This allows the dashboard to run without Telegram.

Backend — backend/scheduler.py

A thin wrapper around APScheduler's AsyncIOScheduler. The sync interval is read from settings.sync_interval_minutes (env var SYNC_INTERVAL_MINUTES, default 5). The scheduler is started in FastAPI's lifespan context manager and stopped gracefully on shutdown.

Backend — backend/main.py

Defines all nine REST endpoints and wires together the application lifecycle. Notable patterns:

  • @asynccontextmanager lifespan — the modern FastAPI approach (replacing deprecated @app.on_event) for startup/shutdown logic.
  • _get_project() — a private helper that does the project lookup and raises HTTPException(404) in one place, keeping all route handlers DRY.
  • Annotated[str | None, Query()] — FastAPI's type-annotated query parameter syntax, which is fully validated and documented in the auto-generated OpenAPI schema at /docs.

Frontend — api/client.ts

A single-file API client that exports all TypeScript interfaces and a typed api object. The generic request<T> function handles error surfacing, and all methods use URLSearchParams to build query strings safely (no manual string concatenation). This means every API call in the app goes through one auditable location.

Frontend — App.tsx

The application shell. Three responsibilities:

  • Theme management — applyTheme() sets/removes document.documentElement.dataset.theme and persists to localStorage. The dark theme is the default (no attribute); light overrides via [data-theme="light"] in CSS.
  • Project selection — a dropdown that fetches the project list and lets the user switch context. The active project ID is lifted up into Shell state and passed down to all pages.
  • Router structure — two routes: / → Dashboard, /:projectId/runs/:runId → RunDetail.

Frontend — Chart Components

All three chart components (SpeedChart, HealthChart, ListingsChart) share the same structural pattern:

  1. Predefined RANGES array with human labels and millisecond durations.
  2. useMemo computes {since, until} ISO strings from the active range or custom date inputs.
  3. useQuery fetches from the API, keyed on [queryKey, projectId, since, until] — any date change automatically triggers a new request.
  4. Recharts ResponsiveContainer + AreaChart renders the data.
  5. Three states: loading (skeleton), empty (text message), data (chart).

Frontend — DatePicker.tsx

A fully custom dropdown calendar built from scratch — no external library. Key implementation details:

  • Week starts Monday — (getDay() + 6) % 7 maps Sunday (0) to position 6, so the grid offset is Monday-based.
  • Outside-click close — a mousedown listener on document checks ref.current.contains(e.target). Only added when open === true and cleaned up on close (prevents listener accumulation).
  • alignRight prop — the "to" picker in a row would overflow the right edge of the viewport if it opened leftward. alignRight switches positioning to right: 0, pinning it to the button's right edge.
  • Sync effect — when value changes externally (e.g., range buttons reset dates), a useEffect updates viewYear/viewMonth so the calendar shows the correct month.

Frontend — LogStream.tsx

Expandable log table with level filtering. Each row is a LogRow that tries to parse log.details as JSON. If parsing succeeds and the object is non-empty, the row becomes clickable and reveals a JsonTree in an expanded row beneath. This pattern avoids loading all JSON at once — only opened rows render the tree.

Frontend — JsonTree.tsx

A recursive component that renders arbitrary JSON. Primitive types (null, boolean, number, string) each have dedicated components with semantic colour coding. Objects and arrays are JsonNode instances that default to open at depth < 2 and collapse beyond that. This is the only component that uses Tailwind utility classes (instead of inline styles) because it predates the design system.

Software Engineering Best Practices

Patterns applied throughout the codebase and why they matter.

Separation of Concerns

The backend is divided into six distinct modules with clear responsibilities. No module reaches into another's domain: metrics.py knows nothing about HTTP; main.py knows nothing about SQL. This makes each module independently testable and replaceable.

ModuleResponsibilityKnows about
config.pyLoad settingsYAML, .env
db.pyData persistenceSQLite, aiosqlite
sheets.pyExternal data ingestionGoogle Sheets API, db.py
metrics.pyAnalytics computationdb.py only
alerts.pyNotificationshttpx, db.py
scheduler.pyBackground timingAPScheduler, sheets.py, alerts.py
main.pyHTTP interfaceAll above modules

DRY (Don't Repeat Yourself)

  • _RUNS_CTE — the complex run-pivot SQL is defined once and reused by both list_runs() and get_run().
  • SELECT_COLS template string — the speed query's column list is defined once, with f-string clauses for conditional where/having/limit logic.
  • _get_project() — project validation and 404 raising is in one function, called by every route.
  • Chart component pattern — all three charts share the same RANGES → useMemo → useQuery → AreaChart structure, reducing cognitive overhead when reading any one of them.
  • api/client.ts — all fetch calls flow through the single generic request<T> function.

Idempotency

The sync pipeline is designed to be safe to re-run at any time without causing data corruption:

  • INSERT OR IGNORE with a content-based dedup index on (project_id, run_id, timestamp, module, function, line) means inserting the same log row twice — or re-syncing after a sheet reset — has no effect.
  • sync_state uses INSERT ... ON CONFLICT DO UPDATE (upsert) keyed on (project_id, sheet_name), so updating the last-synced row for any tab is always safe regardless of prior state.

Type Safety

The entire frontend is TypeScript. Every API response shape has a corresponding interface (Run, Overview, HealthPoint, etc.), so the TypeScript compiler catches mismatches at build time rather than at runtime. The backend uses Python type hints throughout, with list[dict], str | None, and dict[str, Any] return types on every function.

Error Handling

  • Backend: sync_project() wraps the Sheets API call in try/except and re-raises, letting the caller (main.py) convert it to an HTTPException(500).
  • Backend: alerts.py catches all httpx exceptions and logs them without propagating — alert failure should never crash the sync cycle.
  • Frontend: TanStack Query surfaces errors in the { error } field of useQuery — though the current UI doesn't render error states visually, the data is available for future improvement.
  • Frontend: JSON parsing in LogRow and get_filter_breakdown() is wrapped in try/catch / try/except to prevent a single malformed details string from breaking the entire component or query.

Component Composability

React components are small, single-purpose, and accept props rather than reaching into global state. MetricCard renders one KPI from four props. PhaseBreakdown renders a stacked bar from four millisecond values. DatePicker is a pure controlled component — it holds no knowledge of the chart it's used in. This makes each component reusable across the app.

Responsive Design

Layout utilities in index.css are defined as BEM-style utility classes (.oh-grid-4, .oh-chart-grid, .oh-header-inner) with media query breakpoints. This separates layout logic from component logic — a component never needs to know its container width.

✅
One source of truth for layout breakpoints

All responsive rules are in index.css. Any future component that needs a 4-column-to-2-column collapse simply adds className="oh-grid oh-grid-4" — no media queries in the component itself.

Python Best Practices & Idioms

Pythonic patterns, async techniques, and standard library usage.

from __future__ import annotations

Every backend module uses this import. In Python 3.10+ it enables postponed evaluation of annotations — type hints like str | None are stored as strings and evaluated lazily. This means you can use newer union syntax (X | Y) even if the runtime would otherwise reject it, and avoids circular import issues in type annotations.

Dataclasses for Configuration

@dataclass
class ProjectConfig:
    id: str
    name: str
    sheets_id: str
    sheet_name: str
    credentials_file: Path

Dataclasses provide free __init__, __repr__, and __eq__ without boilerplate. Compared to TypedDict or plain dict, they allow dot-access, IDE completion, and type checking. The Path type for credentials_file is more correct than str — it makes the type's intent explicit and enables path operations without casting.

Async Context Manager for Database Connections

@asynccontextmanager
async def get_db():
    async with aiosqlite.connect(DB_PATH) as db:
        db.row_factory = aiosqlite.Row
        yield db

The @asynccontextmanager decorator turns a generator into an async context manager. Every caller writes async with get_db() as db: — the connection is automatically opened and closed around the body. This is the Python-async equivalent of RAII (resource acquisition is initialisation).

Walrus Operator Avoidance (Readability First)

While Python 3.8+ supports the walrus operator (:=), the codebase intentionally avoids it in complex SQL result processing. A clear two-step approach (rows = await ...; return [dict(r) for r in rows]) is preferred over clever one-liners that obscure the data flow.

Type Guards and Optional Chaining

avg_listed = (
    round(listed_row[0]["avg_listed"])
    if listed_row and listed_row[0]["avg_listed"] is not None
    else None
)

Rather than trying to call round() on a potentially-None value and catching TypeError, the code explicitly guards both the row's existence and the value's non-nullness. This is Pythonic: explicit is better than implicit.

Statistics Module for In-Process Aggregation

import statistics

speeds = [r["ms"] for r in speed_rows if r["ms"] is not None]
speed_stats = {
    "avg_ms":    round(statistics.mean(speeds)),
    "median_ms": round(statistics.median(speeds)),
}

The standard library's statistics module provides correct, readable mean and median. For small lists (a few hundred run times), this is far simpler than adding dependencies like NumPy or computing everything in SQL.

ZoneInfo for Timezone-Correct Date Arithmetic

from zoneinfo import ZoneInfo, ZoneInfoNotFoundError

try:
    zone = ZoneInfo(tz)
except ZoneInfoNotFoundError:
    zone = timezone.utc

midnight_local = datetime.now(zone).replace(
    hour=0, minute=0, second=0, microsecond=0
)
today = midnight_local.astimezone(timezone.utc).strftime("%Y-%m-%dT%H:%M:%S.000Z")

ZoneInfo (Python 3.9+) is the modern, standard-library way to handle IANA timezone data. The pattern: get local midnight, convert to UTC, format as ISO string. The fallback to timezone.utc on unknown timezone strings makes the function robust against client-sent bogus values.

List Unpacking and Spread in Parameters

rows = await db.execute_fetchall(
    sql, [project_id, run_id, *levels]  # spread levels into param list
)

Using *levels inside a list literal to build a flat parameter array is idiomatic Python — far cleaner than building the list with concatenation.

f-strings with Conditional Clauses

filter_extra = "AND started_at <= ?" if until else ""
params = [project_id, since]
if until:
    params.append(until)

sql = f"""
SELECT ... FROM with_rates
WHERE started_at >= ? {filter_extra}
"""

Conditional SQL clauses are built by computing a string fragment early and embedding it in the f-string. This avoids deeply nested if/else branching or string concatenation. The parameter list is built in parallel, keeping positions consistent.

Performance Optimisations

Where the system is fast, why, and where the headroom is.

SQLite Indices

Four indices are created at startup covering every production query access pattern:

IndexColumnsQueries served
idx_logs_project_run(project_id, run_id)Run detail, run logs, all per-run aggregations
idx_logs_project_ts(project_id, timestamp)Time-range filtering in metrics
idx_logs_level(project_id, level)Health rate, warning/error counts
idx_logs_alert(project_id, alert_sent, level)Unsent alert query (covering index)

The alert index is a covering index — the query WHERE project_id = ? AND alert_sent = 0 AND level IN (...) can be answered entirely from the index without touching the main table rows.

Incremental Sync

Every sync call transfers only new rows from each Google Sheets tab (tracked by row number per tab in sync_state). A project with 10 000 historical log lines spread across multiple rotated tabs doesn't pay the cost of re-reading them on every sync. Only the delta — typically 20–100 rows per run — is transferred, parsed, and inserted. Content-based deduplication provides an additional safety net: even if last_row is reset or a tab is cleared and reused, no duplicate rows enter the database.

Batch Insert with executemany

await db.executemany(
    "INSERT OR IGNORE INTO logs ...",
    rows  # list of dicts — all rows in one round-trip
)

A single executemany call wraps all new rows in one database transaction, avoiding the overhead of per-row BEGIN/COMMIT cycles.

React Query Caching

TanStack Query caches every API response by its query key. Multiple components asking for the same data (e.g., both the overview and chart components running simultaneously) will share one cached result — no duplicate network requests. The refetchInterval: 60_000 setting keeps the dashboard live without the user having to manually refresh.

useMemo for Derived State

const { since, until } = useMemo(() => {
  if (range === "Custom") {
    return { since: customFrom ? toISODate(customFrom) : undefined, ... };
  }
  const ms = RANGES.find((r) => r.label === range)!.ms;
  return { since: new Date(Date.now() - ms).toISOString(), until: undefined };
}, [range, customFrom, customTo]);

useMemo ensures the since/until strings are only recomputed when the relevant state changes. Without it, every render would recompute the date strings and potentially trigger unnecessary query refetches.

Dynamic SQL Template Strings

The SELECT_COLS pattern in get_speed_series() avoids duplicating a 15-line SQL select list for the since/until vs limit cases. At a performance level, this is negligible — but it also reduces the chance of the two paths diverging silently, which would be a correctness bug masquerading as a performance issue.

requestAnimationFrame for Enter Animations

useEffect(() => {
  const id = requestAnimationFrame(() => setAnimated(true));
  return () => cancelAnimationFrame(id);
}, []);

FilterBreakdown and PhaseBreakdown use requestAnimationFrame to trigger their CSS transitions on the next paint frame. Setting animated state synchronously in useEffect would race with the browser paint and may not show the animation. RAF defers until the browser is ready to paint, guaranteeing a visible transition.

Window Function Computation at the Database Layer

Computing cumulative rates entirely in SQL — rather than fetching all rows and aggregating in Python — keeps the data transfer minimal. For a project with 2 000 runs, the database returns at most 2 000 rows for the health chart, each with two pre-computed percentages. A Python loop over all log entries would be orders of magnitude more expensive.

Product Development Lessons

What the iterative development process revealed about building tools you use yourself.

01

A metric that can't be compared to something else is almost useless

The original health chart showed per-hour bucket rates (e.g., "20% of runs in the 5pm hour had warnings"). That number is hard to act on in isolation. The breakthrough was realising the summary card number — the all-time rate — is the benchmark every user is already anchored to. Changing the chart to show cumulative trajectory made both values coherent and mutually reinforcing.

02

The last data point on a chart should equal the summary number

If a dashboard shows "Warning rate: 8.9%" in a card, and the chart line ends at 11.2%, users lose trust in both numbers. Investing in making these two sources agree — via the outer-WHERE fix in the SQL window function — is not just a technical fix; it's a trust restoration that makes the whole dashboard credible.

03

Remove features that don't earn their space

The Pipeline Funnel chart was removed when it became clear that the same data was visible in the run table's "Funnel" column. Every UI element competes for attention. Removing the funnel made the Speed Chart more prominent and the page less cluttered. Good product design is as much about deletion as addition.

04

Native inputs break your design system

The native <input type="date"> rendered differently on every OS, ignored all custom CSS, and clashed visually with the rest of the dashboard. The custom DatePicker dropdown took ~200 lines but delivered a component that matches the design system exactly, handles min/max constraints, and behaves the same across all browsers. For user-facing tools you control end-to-end, owning the interaction is often worth the cost.

05

Filter-ability is a first-class feature, not an afterthought

Adding run-level filter buttons (All / OK / Warn / Error / Critical) with live counts turned the runs table from a data dump into an actionable triage interface. Users can go directly to "show me all error runs" in one click. Design filtering into your data tables from the start.

06

Consistency in UI state matters more than any individual component

When some filter chips were highlighted and others weren't, users couldn't tell if the highlighting was meaningful or accidental. Making all chips use the same base style — with only the active chip's count badge changing colour — eliminated the confusion. Inconsistent visual states create cognitive noise even when the underlying data is correct.

07

The most useful sort order is usually not chronological

Rejected gigs sorted by timestamp are a random walk. Sorted by fee descending, the most financially significant rejections appear first — immediately answering "am I losing well-paid opportunities to this filter?" A simple sort comparator changed the table from a log into a decision-support tool.

08

Auto-refresh is the difference between a dashboard and a report

The 60-second refetchInterval on all queries means the dashboard stays current without user action. This is the defining characteristic of an operational dashboard vs a static report — you can leave it open on a second monitor and trust it always shows the current state.

Concepts Glossary

Technical terms used in the codebase, explained plainly.

aiosqlite
An async wrapper around Python's built-in sqlite3 module. It offloads database I/O to a thread pool so SQLite queries don't block FastAPI's event loop.
APScheduler
Advanced Python Scheduler — a job scheduling library. The AsyncIOScheduler variant integrates with Python's asyncio event loop, allowing scheduled async coroutines without a separate thread.
Common Table Expression (CTE)
A named temporary result set defined with WITH name AS (...) before the main query. Useful for breaking complex queries into readable steps, and for referring to the same derived table multiple times.
Conditional Aggregation
Using MAX(CASE WHEN condition THEN value END) inside a GROUP BY query to pivot multiple log rows for the same run_id into a single summary row with multiple fields.
Covering Index
A database index that contains all columns referenced by a query, so the database can answer the query entirely from the index without reading the main table. Eliminates a secondary lookup step.
CSS Custom Properties
Variables defined in CSS using --name: value syntax and referenced with var(--name). Changing one property on :root or a parent element cascades to all children that use that variable — the foundation of the theming system.
Idempotent
An operation that produces the same result whether it is applied once or many times. INSERT OR IGNORE is idempotent — inserting the same row repeatedly is harmless.
Incremental Sync
Fetching only new data since the last successful sync, rather than re-fetching the entire dataset. The sync_state table records the last synced row number per sheet tab (project_id + sheet_name composite key) to make this possible across multiple rotated log tabs.
Lifespan Context Manager
FastAPI's mechanism for startup/shutdown logic. Code before yield runs on startup; code after runs on shutdown. Replaces the deprecated @app.on_event("startup") pattern.
requestAnimationFrame
A browser API that schedules a callback to run just before the next repaint. Used in React components to trigger CSS animations on mount — guarantees the element has been painted at its initial state before the transition begins.
Row Factory
A function assigned to db.row_factory that transforms raw database rows. Setting it to aiosqlite.Row makes rows dict-like: accessible by column name (row["column"]) instead of numeric index.
Service Account
A Google Cloud identity used by applications (not users) to access APIs. The dashboard uses a service account JSON key to authenticate with Google Sheets API without user interaction.
TanStack Query
A React library for managing server state. It provides automatic caching, background refetching, loading/error states, and query invalidation — replacing manual useEffect + fetch patterns.
useMemo
A React hook that memoizes a computed value, recalculating it only when its declared dependencies change. Prevents unnecessary recomputation on every render.
URLSearchParams
A browser/Node.js API for building URL query strings. Handles encoding special characters and & separators automatically — safer than manual string concatenation.
Upsert
An operation that inserts a row if it doesn't exist, or updates it if it does. SQLite syntax: INSERT INTO ... ON CONFLICT(pk) DO UPDATE SET .... Used for sync_state to atomically update the last-synced row.
Vite Proxy
A Vite dev server feature that forwards requests matching a path prefix to another server. The '/api': 'http://localhost:8000' proxy lets the React frontend make /api calls without CORS issues during development.
Window Function (SQL)
A SQL function that computes a value across a set of rows related to the current row, without collapsing them into a group. SUM(...) OVER (ORDER BY ts ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) is a running total — the value at each row is the sum of all preceding rows including itself.
ZoneInfo
Python's standard library (3.9+) for IANA timezone database access. Replaces the pytz third-party library. Used to correctly compute "midnight in the user's local timezone" for the "runs today" metric.

Key Takeaways for Career Growth

What building this project teaches you about engineering at a higher level.

01

SQL window functions are the single most underused professional skill

Most developers reach for Python/JavaScript to compute running totals, moving averages, and rankings — but SQL can do this with zero data transfer to the application layer. Knowing window functions (SUM OVER, ROW_NUMBER OVER, LAG, LEAD, RANK) puts you in the top 10% of SQL practitioners and solves problems that would otherwise require complex application code or specialised analytics tools.

02

Build the monitoring layer before you need it

This dashboard was built because organist_bot was already running blindly. In professional settings, observability (logging, metrics, tracing) should be designed alongside the feature, not retrofitted after problems emerge. The cost of instrumentation is low when you're building; the cost of debugging a production incident without it is very high.

03

Async Python is a career-differentiating skill

FastAPI with async/await and aiosqlite represents modern Python backend development. Understanding event loops, coroutines, context managers, and how to combine sync and async code without blocking the event loop is increasingly required for high-performance Python services at scale.

04

TypeScript pays dividends at refactoring time

When the HealthPoint interface changed (dropping hour, adding started_at), the TypeScript compiler immediately flagged every component that consumed the old shape. Without types, this would have been a silent runtime bug. Strong types are not overhead — they're free documentation and an automated regression test for your data contracts.

05

Separation of concerns is the foundation of maintainability

The clean module boundaries in this codebase mean that swapping Google Sheets for a Webhook API requires changes only to sheets.py and scheduler.py — nothing in metrics.py, main.py, or the frontend changes. Professional codebases that don't have this separation become unmaintainable. The discipline of asking "does this module need to know about X?" is a habit worth cultivating from day one.

06

Server state management is a solved problem — use the library

Implementing useEffect + fetch + useState for async data is error-prone: race conditions, stale closures, missing loading states, and no caching. TanStack Query solves all of this. Knowing which problems have canonical library solutions — and reaching for those libraries instead of re-implementing them — is a hallmark of senior engineering.

07

The feedback loop between using your own tool and improving it is irreplaceable

Every iteration of this dashboard was driven by using it and noticing something that felt wrong: the Y-axis stopping at 100% when all rates were under 5%. The "00:00" timestamps cluttering the listings X-axis. The filter buttons that were sometimes highlighted for no apparent reason. Dogfooding — using what you build — produces product insights that no specification document can anticipate. Build tools you actually use.

08

Incremental delivery beats big-bang releases

This dashboard was built one chart, one metric, one fix at a time — each iteration immediately testable and useful. The temptation in personal projects is to plan a "complete" version before writing any code. The reality is that the most valuable insights about what to build come from the act of using an incomplete version. Ship early, iterate fast.

09

Configuration and secrets management is non-negotiable from day one

Using python-dotenv and a projects.yaml config file from the start means: (a) no hardcoded credentials in source code, (b) easy environment switching (dev/prod), and (c) the project is git-safe by design. Professional environments enforce this at the CI level. Building the habit in personal projects means it becomes natural.

10

Polished UX is an engineering skill, not just a designer's job

The animated bar chart transitions, the DatePicker calendar, the smooth theme crossfade, the colour-coded timing indicators in the runs table — none of these required a designer. They required understanding CSS transitions, requestAnimationFrame, and thinking carefully about what information each visual element communicates. Engineers who care about polish ship products that people trust and enjoy using. That trust translates directly into whether a tool gets adopted or abandoned.
